   env seam (in the two upload scripts) and the pytest harness that 003 and 004 reuse.
3. **003** and **004** — the two data-integrity bug fixes, each with a regression test on
   the 002 harness. 003 and 004 both touch `daily_fantasy_log_upload.py` in different
   sections; do 003 before 004 to avoid overlapping edits, though they're independent.

   fixes so their tests are already green and act as a safety net for the refactor.

   logic; 007 centralizes season filters. Same files, so sequence them.

   alone. Schedule any time after 005.

## Findings considered and rejected (so they aren't re-audited)

- **SQL string interpolation in IN-lists** (export scripts, `run_db_patch.py`): player
  names are DB-sourced and single-quote-escaped (`.replace("'", "''")`) before
  interpolation; table/column names in the patch script are hardcoded constants. Not
  meaningfully exploitable here. Plan 006 keeps the existing escaping via `to_sql_in_list`.
- **Performance**: data volumes are tiny (hundreds of players, low-thousands of logs).
  Full-table pandas loads and O(n·m) fuzzy matching are fine; no real wins. Not worth doing.
- **Credential hygiene**: `.env`, `token.json`, `client_secrets.json` are all git-ignored
  and nothing sensitive is committed. Clean — no action.
- **Duplicate `os.makedirs(PROCESSED_FOLDER, exist_ok=True)` lines / dead commented code**
  at the end of `daily_player_upload.py`: trivial; fold into another change if convenient,
  not worth a dedicated plan.
- **Misleading orchestrator filename** (`daily_fantasy_log_upload.py` is the orchestrator):
  documented in `CLAUDE.md`; renaming would break the documented invocation. Not worth it.

## New findings considered and rejected (plans 010–012 audit)

- **Structured logging (`logging` module)** — All scripts use `print()`. Replacing with
  `logging` would improve post-run diagnostics for headless scheduled runs (CONCERNS.md:12).
  Effort L (all modules), low risk, but low urgency since the email notification already
  delivers a summary. Deferred — not worth a plan until the scheduling moves off Windows
  Task Scheduler.
- **CSV export accumulation** — `csv_exports/` grows by 2+ files per day with no cleanup.
  Minor operational concern; the files are small. Not worth a plan (trivial manual cleanup
  or a simple `glob.glob` + `os.remove` one-liner if needed).
- **`L30_DK_POINTS_sum`/`L30_MINUTES_sum` extra columns in `fantasy_averages`** — The
  rename map in `create_summary_tables.py` doesn't rename these intermediate aggregation
  columns; they persist in the table. Cosmetic; downstream queries ignore them. Not worth
  a dedicated plan.
- **Direction: migrate daily trigger to GitHub Actions** — The Windows Task Scheduler
  dependency (CONCERNS.md:15) could be replaced with a scheduled GH Actions workflow
  (`on: schedule: cron`). Deferred: requires secrets management for the Google Drive
  token and Gmail credentials, which is a meaningful scope change.
