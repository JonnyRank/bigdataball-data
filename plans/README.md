## Findings considered and rejected (so they aren't re-audited)

- **SQL string interpolation in IN-lists** (export scripts, `run_db_patch.py`): player
  names are DB-sourced and single-quote-escaped (`.replace("'", "''")`) before
  interpolation; table/column names in the patch script are hardcoded constants. Not
  meaningfully exploitable here. Plan 006 keeps the existing escaping via `to_sql_in_list`.
- **Performance**: data volumes are tiny (hundreds of players, low-thousands of logs).
  Full-table pandas loads and O(n·m) fuzzy matching are fine; no real wins. Not worth doing.
- **Credential hygiene**: `.env`, `token.json`, `client_secrets.json` are all git-ignored
  and nothing sensitive is committed. Clean — no action.
- **Duplicate `os.makedirs(PROCESSED_FOLDER, exist_ok=True)` lines / dead commented code**
  at the end of `daily_player_upload.py`: trivial; fold into another change if convenient,
  not worth a dedicated plan.
- **Misleading orchestrator filename** (`daily_fantasy_log_upload.py` is the orchestrator):
  documented in `CLAUDE.md`; renaming would break the documented invocation. Not worth it.
  **Superseded 2026-07-24 by plan 020** — rather than rename, the orchestration is being
  extracted into a new `daily_pipeline.py`, leaving `daily_fantasy_log_upload.py` as
  fantasy-log ingestion only (parallel to `daily_player_upload.py`).

  (`on: schedule: cron`). Deferred: requires secrets management for the Google Drive

## New findings considered and rejected (deep audit 2026-07-24)

- **`player_logs` box-score dedup lacks the Int64/int-cast hardening `fantasy_logs`
  got in plan 014** — `daily_player_upload.py:91-95,210-211` builds `log_key` from
  `PLAYER_ID.astype(str)` without normalizing the DB-side storage affinity (as the
  hardened `daily_fantasy_log_upload.py:146-153` now does) and without casting incoming
  IDs to int. If `player_logs.PLAYER_ID` ever stored as REAL, dedup would mis-key (a
  DB `"12345.0"` vs incoming `"12345"`), and with plan 012's UNIQUE index the mismatch
  would fail the whole file with `IntegrityError` rather than silently duplicate.
  **Recorded as investigate-only, not planned**: the player feed appears to deliver clean
  integer IDs (unlike the DFS feed that motivated plan 014), so this asymmetry may never
  trigger. Promote to a plan only if an `IntegrityError` on `player_logs` is actually
  observed. (LOW confidence on whether it can fire; the fix, if needed, mirrors plan 014.)
- **Doc drift: `STACK.md` says CI pins Python 3.11** but `.github/workflows/test.yml` now
  runs 3.13 (and `pyproject.toml` floor is `>=3.11`). One-line fix; fold into any future
  doc edit rather than a dedicated plan. (Plan 018 already tells its executor to update the
  STACK.md "Linting / Formatting" note if the Ruff-defaults claim turns out stale.)
