# INTEGRATIONS

## Google Drive (data source)

- **Library:** `google-api-python-client` (`googleapiclient.discovery.build("drive", "v3", ...)`), `google-auth-oauthlib`, `google-auth`.
- **Auth:** 3-legged OAuth 2.0 installed-app flow, **interactive** (`auth_manager.py`). First run opens a browser (`flow.run_local_server(port=0)`); the resulting token is written to `token.json` and reused/refreshed on later runs. **Cannot run headless** without a pre-existing valid `token.json`. Scope: `https://www.googleapis.com/auth/drive.readonly` (`config.py:29`).
- **Credentials:** `client_secrets.json` (downloaded from Google Cloud Console) + `token.json` (auto-generated). Both **git-ignored** (`.gitignore:4-5`).
- **What it does:** `drive_ingestion.py` iterates `config.DATASET_JOBS`, queries each Drive folder for files whose name contains a match substring, sorts by `createdTime`, and downloads the latest into the matching local folder (`drive_ingestion.py:15-95`). `supportsAllDrives=True` / `includeItemsFromAllDrives=True` are set for Shared Drives.
- **Jobs (`config.py:10-24`):** "DFS Feed" (`-dfs-feed.xlsx` â†’ `Daily_Fantasy_Logs/`) and "Player Feed" (`season-player-feed.xlsx` â†’ `Daily_Player_Logs/`). Folder IDs come from env vars `DRIVE_FOLDER_ID_DFS` / `DRIVE_FOLDER_ID_PLAYER`.
- **Download root:** `config.BASE_DOWNLOAD_DIR = r"G:\My Drive\Documents\bigdataball"` â€” **hardcoded, no fallback**, so Drive ingestion requires the `G:` mount even though the rest of the pipeline can fall back to local `Data/`.

## Local SQLite Database

- **File:** `nba_fantasy_logs.db` under the resolved base path (git-ignored via `*.db`; ~36 MB locally). Regenerated by scripts, never committed.
- **Access:** SQLAlchemy + pandas in pipeline scripts; raw `sqlite3` in maintenance scripts. Connection string `sqlite:///{DB_PATH}`.
- **No external/managed DB** â€” purely local file.

## DraftKings (slate input)

- **Not an API.** Export scripts read a user-supplied `DKEntries.csv` from `~/Downloads` (`os.path.expanduser("~") / "Downloads" / "DKEntries.csv"`). Header row is auto-detected by scanning the first 50 lines for `"Position"` + `"Name + ID"` (`export_slate_averages_vw.py:29-46`). DK player names are fuzzy-matched to DB names (`thefuzz`, score â‰Ą 90).

## Email Notification (Gmail SMTP)

- **Library:** stdlib `smtplib.SMTP_SSL("smtp.gmail.com", 465)` + `email.message.EmailMessage` (`email_notifier.py`).
- **Auth:** `EMAIL_SENDER` + `EMAIL_PASSWORD` (a Gmail **app password**), recipient `EMAIL_RECEIVER` â€” all from env. Toggled by `config.EMAIL_ENABLED` (default `True`, `config.py:32`).
- **What it sends:** end-of-run success/error summary; if DK players went unmatched, a warning email plus an append to `todo_mappings.txt` (`daily_fantasy_log_upload.py:355-395`).

## Environment Variables (`.env`, loaded by `python-dotenv` in `config.py`)

No `.env.example`/`.env.template` is committed; required vars are read in `config.py`:

| Var | Used for |
|-----|----------|
| `DRIVE_FOLDER_ID_DFS` | Drive folder ID for the DFS feed (`config.py:13`) |
| `DRIVE_FOLDER_ID_PLAYER` | Drive folder ID for the player feed (`config.py:20`) |
| `EMAIL_SENDER` | Gmail sender address (`config.py:33`) |
| `EMAIL_PASSWORD` | Gmail app password (`config.py:34`) |
| `EMAIL_RECEIVER` | Notification recipient (`config.py:35`) |
| `BIGDATABALL_DATA_DIR` | Optional override for the base data path (read in upload scripts + `check_ingest_duplicates.py`; used by tests) |

`.env`, `client_secrets.json`, `token.json` are all git-ignored.

## CI / Automation

- **GitHub Actions** (`.github/workflows/`):
  - `test.yml` â€” installs deps, runs `pytest -q` on push to `main` and on PRs (ubuntu, Python 3.11).
  - `claude.yml`, `claude-auto-pr-review.yml` â€” Claude Code GitHub Action workflows (tooling, not pipeline).
- The daily pipeline is triggered by **Windows Task Scheduler** on the maintainer's machine (runs `daily_fantasy_log_upload.py`). This is host-side configuration, not committed to the repo â€” the scheduled task relies on a pre-existing valid `token.json` so the OAuth step doesn't block.
- **Claude Code on the web** auto-bootstraps a project-local `.venv` (installing `requirements.txt` + `requirements-dev.txt`) via the `SessionStart` hook in `.claude/hooks/session-start.sh`, registered in `.claude/settings.json`. It is remote-only (guards on `CLAUDE_CODE_REMOTE`), so local sessions are untouched. Network access stays at the default **Trusted** level (PyPI is allowlisted); no environment variables or secrets are required to run the test suite, which stubs the Google/SMTP integrations. `.claude/settings.json` also pre-allowlists the claude-code-remote MCP tools `send_later` and `delete_trigger` (both server-name spellings), so web sessions that monitor PRs can schedule and clean up self check-ins without permission prompts.
